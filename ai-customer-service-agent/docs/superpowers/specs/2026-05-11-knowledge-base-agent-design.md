# AI 智能知识库问答 Agent — 产品设计文档

## 目标

在现有 ai-customer-service-agent 项目基础上，新增一个**智能知识库问答 Agent** 子系统，用于展示 AI Agent 产品设计与落地能力，面向 AI Agent 产品经理/产品运营暑期实习求职。

## 产品定位

面向团队/企业的智能知识库问答 Agent。用户上传文档后，用自然语言提问，Agent 能**拆解复杂问题、跨文档推理、自主选择检索工具、给出带出处的答案**。

和简单 ChatGPT 的区别：答案**有据可查**，每条结论都能追溯到原始文档的哪一页哪一段，或网络来源的哪个网页。

## 核心用户流程

```
上传文档（PDF/Word/TXT/网页链接）
        ↓
系统自动解析 → 切片 → 向量化 → 入库
        ↓
用户用自然语言提问
        ↓
Agent 判断问题类型 & 所需工具
        ↓
   ┌────────────────┬──────────────────┐
   ↓                ↓                  ↓
知识库能回答    知识库不够        需要实时信息
   ↓                ↓                  ↓
RAG 检索     调用外部搜索 API    调用搜索 API
   ↓                ↓                  ↓
返回带引用答案  返回带网络来源答案  返回带网络来源答案
```

### 关键体验设计

1. **溯源高亮**：答案中每个结论后面有 `[来源]` 标签，点击后高亮原文对应段落或跳转网页链接
2. **拆解过程可见**：复杂问题时展示 Agent 思考过程（"这个问题我拆成 3 个子问题..."），让 Agent 行为透明可解释
3. **优雅的"不知道"**：当知识库和网络搜索都没有相关信息时，明确告知用户而非编造答案
4. **来源类型区分**：`[知识库]` 和 `[网络]` 标签用不同颜色区分，用户一眼能判断信息可信度

## 功能清单

### 核心功能

| 功能 | 说明 |
|------|------|
| **知识库管理** | 创建/删除知识库，上传文档（PDF/Word/TXT/网页链接），查看文档列表和解析状态 |
| **智能问答** | 自然语言提问，Agent 自动判断走知识库还是外部搜索，流式返回答案 |
| **答案溯源** | 每条结论标注来源（文档名+段落 vs 网页链接），点击可查看原文 |
| **思考过程可见** | 展示 Agent 的推理步骤（问题拆解 → 工具选择 → 检索 → 整合） |
| **外部搜索集成** | 知识库不足时自动调用搜索 API（Tavily），结果标注网络来源 |
| **对话历史** | 保存历史问答记录，支持查看和继续对话 |
| **反馈机制** | 用户对答案点 👍/👎，用于展示"怎么评估 Agent 质量"的产品思维 |

### 不做

- 用户登录/权限系统（Demo 不需要，单用户即可）
- 实时协作
- 自定义 Agent 编排（另一个产品方向，先聚焦）

## 架构设计

### 三层架构

```
┌─────────────────────────────────────────────┐
│               前端展示层                      │
│   Next.js（复用现有项目框架）                  │
│   知识库页面 │ 问答页面 │ 对话历史 │ 反馈    │
│   设计：调用 frontend-design skill，高级      │
│   视觉效果，避免通用 AI 模板风格              │
└──────────────────┬──────────────────────────┘
                   │ HTTP + SSE 流式
┌──────────────────┴──────────────────────────┐
│               Agent 编排层                    │
│   FastAPI (Python)，在现有 backend/ 扩展      │
│   ├─ 问题分析 & 拆解 (Agent Planner)         │
│   ├─ 工具调度 (Tool Router)                  │
│   ├─ 答案生成 & 引用整合 (Answer Builder)    │
│   └─ 反馈收集 (Feedback Collector)           │
└──────────────────┬──────────────────────────┘
                   │
┌──────────────────┴──────────────────────────┐
│               数据 & 工具层                   │
│   ├─ 文档解析：pdfplumber / python-docx      │
│   ├─ 向量存储：ChromaDB（轻量，免部署）       │
│   ├─ 向量模型：text-embedding-3-small (API)  │
│   ├─ LLM：OpenAI 兼容 API（复用现有集成）     │
│   ├─ 搜索 API：Tavily（AI Agent 专用搜索）    │
│   └─ 数据库：SQLite（对话历史 + 反馈）        │
└─────────────────────────────────────────────┘
```

### 技术选型理由

| 选择 | 理由 |
|------|------|
| **ChromaDB** | 开箱即用，pip install 即可，不需要单独部署服务，Demo 场景足够 |
| **SQLite** | Zero setup，存对话历史和反馈数据够用 |
| **复用 Next.js + FastAPI** | 现有项目就是这个架构，直接在 ai-customer-service-agent 上扩展 |
| **Tavily Search API** | 专为 AI Agent 设计的搜索 API，返回结构化结果，比传统搜索引擎更适配 LLM |
| **text-embedding-3-small** | OpenAI 嵌入模型，便宜（$0.02/1M tokens）且效果好，API 调用简单 |
| **混合检索（BM25 + 向量）** | 避免纯向量检索漏掉精确匹配（如文档编号、日期、人名），互补短板 |

### 关键设计决策

1. **混合检索**：关键词 BM25 + 向量余弦相似度，结果合并去重，交集优先
2. **流式输出**：答案逐字生成（SSE），但思考过程用结构化 JSON 步骤展示，不与答案混在一起
3. **引用格式统一**：不管知识库还是网络来源，统一用 `[[来源ID]]` 标记，前端解析渲染为可点击链接
4. **Agent 工具选择逻辑**：先检索知识库 → 相关性分数低于阈值（0.7）→ 自动触发外部搜索 → 两路结果合并后一起给 LLM 生成答案

## 数据流

### 文档入库流程

```
用户上传文档 → 后端接收文件 → 文档解析提取文本
                                    ↓
                              文本切片（chunk_size=512, overlap=50）
                                    ↓
                              调用 embedding API 生成向量
                                    ↓
                              文本 + 向量存入 ChromaDB collection
                                    ↓
                              返回入库成功 + 文档列表更新
```

### 问答流程

```
用户提问 → Agent Planner 分析问题类型
                ↓
          Tool Router 决策：知识库 / 搜索 / 两者
                ↓
          ┌─ 知识库检索：BM25 + 向量 → 取 top 5 → 重排序 → top 3
          └─ 外部搜索：Tavily API → 取 top 3 结果 → 提取文本
                ↓
          Answer Builder：拼接检索结果 + 用户问题 → LLM 生成答案
                ↓
          SSE 流式返回：先发思考步骤 JSON → 再发答案 token 流 → 最后发引用列表
```

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/knowledge-bases` | 创建知识库 |
| GET | `/api/knowledge-bases` | 列出所有知识库 |
| DELETE | `/api/knowledge-bases/{id}` | 删除知识库 |
| POST | `/api/knowledge-bases/{id}/documents` | 上传文档 |
| GET | `/api/knowledge-bases/{id}/documents` | 列出文档 |
| DELETE | `/api/knowledge-bases/{id}/documents/{doc_id}` | 删除文档 |
| POST | `/api/knowledge-bases/{id}/chat` | 提问（SSE 流式返回） |
| GET | `/api/knowledge-bases/{id}/history` | 获取对话历史 |
| POST | `/api/chat/{message_id}/feedback` | 提交反馈（👍/👎） |

## 前端页面

在现有 Next.js 项目中新增以下页面：

| 页面 | 路由 | 说明 |
|------|------|------|
| 知识库列表 | `/knowledge-bases` | 展示所有知识库，创建/删除 |
| 知识库详情 | `/knowledge-bases/[id]` | 文档管理 + 问答界面 |
| 对话历史 | `/knowledge-bases/[id]/history` | 历史对话列表，可继续对话 |

前端设计调用 `frontend-design` skill，要求：
- 避免通用 AI 模板风格（不要白底蓝按钮的 ChatGPT 风格）
- 有设计感，配色和排版有辨识度
- 思考过程展示用卡片式步骤，不用纯文本框
- 溯源引用用不同颜色标签区分知识库来源和网络来源

## 面试准备清单

本项目设计时已考虑以下面试高频问题：

| 面试官问 | 项目中的对应实践 |
|----------|-----------------|
| RAG 是什么，怎么优化？ | 混合检索（BM25+向量）、切片策略、重排序——全在项目里做过 |
| Agent 和普通 LLM 的区别？ | Agent 会主动决策——选工具、拆问题、验证答案，项目中 Tool Router 和 Agent Planner 体现了这一点 |
| 怎么降低幻觉？ | 溯源机制 + 低置信度触发搜索 + 明确告知"不知道" |
| 怎么评估 Agent 质量？ | 用户反馈（👍/👎）+ 引用准确率 + 回答覆盖率 |
| 为什么选 ChromaDB 而不是 Milvus/Pinecone？ | 免部署、pip install 即可、Demo 场景够用，生产环境会换 |
| 产品经理思维体现在哪里？ | 用户流程设计、信任感设计（溯源）、MVP 范围取舍、反馈闭环 |
